# Component notes ‚Äî RAKwireless RAK4631 (WisBlock)

**nRF52840** + on-board **SX1262** (no FEM) on a **RAK19007** base board, with a
**RAK1921** 0.96‚Ä≥ SSD1306 OLED. Built with `-DDEVICE_RAK4631`. This is the
project's first **cross-MCU** target ‚Äî not an ESP32 ‚Äî so it exercises the
`platform::`/`kv::` portability seams hardest; most of the lessons below are
really **nRF52-family** lessons it shares with the Wio Tracker L1
(`-DDEVICE_WIO_TRACKER_L1`). The radio is covered in `sx1262.md`; full pin
provenance is in `reference/rak4631/` and `RAK-port-plan.md` ¬ß2.
Hardware-validated on a real RAK4631 + RAK19007 + RAK1921 (station 26).

## What's different from the ESP32 boards

The whole nRF52 backend is selected at compile time and lives behind the
portability seams, so the app code (`main.cpp`, `radio.cpp`, `display.cpp`, ‚Ä¶) is
shared:

| Concern | ESP32 (Heltec/Cardputer) | nRF52 (RAK4631 / Wio) |
|---|---|---|
| Core / SDK | arduino-esp32 | Adafruit nRF52 + **SoftDevice S140** |
| BLE stack | NimBLE | **Bluefruit** (`ble_provision_nrf52.cpp`) |
| Persistence | NVS / Preferences | **LittleFS** (`kv_nrf52.cpp`) |
| Radio SPI | HSPI | **global SPI** |
| Flash tool | esptool | **nrfutil DFU** (1200 bps touch) |
| Platform glue | `platform_esp32.cpp` | `platform_nrf52.cpp` |

## Pin map (`src/pins.h` ‚Üê `platformio.ini`)

Pins come from `[env:rak4631]` build flags (`-DPIN_*`); `pins.h` only consumes
them with `#ifndef` fallbacks. The RAK4631's variant pin map **is identity**
(logical == absolute nRF GPIO), unlike the Wio's.

| Function | GPIO | Notes |
|---|---|---|
| SX1262 NSS / SCK / MISO / MOSI | 42 / 43 / 45 / 44 | on-board, global SPI |
| SX1262 NRST / BUSY / DIO1 | 38 / 46 / 47 | **BUSY=46** confirmed vs datasheet (P1.14); some variant.h copies wrongly use 39 |
| **SX126X_POWER_EN** | 37 | **must be driven HIGH before radio init** ‚Äî powers the SX1262 LDO (see below) |
| SX1262 DIO2 / DIO3 | ‚Äî | DIO2 = RF switch (`setDio2AsRfSwitch`), DIO3 = TCXO (1.8 V) |
| OLED SDA / SCL | default `Wire` | RAK1921 SSD1306, **no reset line**, no VEXT gate (RAK19007 always powers it) |
| I2S sidetone ‚Üí MAX98357A (BCLK / LRCK / DIN) | 17 / 31 / 16 | RAK19007 solder headers J10/J11; see audio note |
| Battery VBAT (AIN, analog) | ‚Äî | divider on the RAK19007; `DIVIDER_RAK` **needs bench calibration** (see TODO) |
| **WB_IO2** | 34 | **NOT a free GPIO** ‚Äî it gates the 3V3_S sensor rail; driving it cuts/garbles that supply |
| Telegraph key / mode button | unconfirmed | `PIN_KEY`/`PIN_MODE_BTN` not yet validated on HW (menu reached via 5 s idle auto-select) |

Free header GPIOs after radio + I2C OLED + VBAT are only **15 / 16 / 17 / 31**
(WB_IO5=9 / WB_IO6=10 are NFC pins needing a one-time UICR fuse-write). No FEM:
do **not** define `HAS_FEM` ‚Äî the `pa`/`lna` commands report "no FEM on this
board," and the SX1262 chip output (+22 dBm) **is** the antenna power.

## Lessons learned

### SX126X_POWER_EN (GPIO37) must go HIGH before radio init

Unlike the Heltec (whose SX1262 is always powered), the RAK gates the radio LDO
behind GPIO37. `radio.cpp` drives it HIGH before `beginFSK()`; miss it and init
fails. This is the RAK analogue of the Heltec's FEM power-on, minus the FEM.

### Two boot-hang bugs cost the most during bring-up

Both made the node **silently** hang ‚Äî dead serial, OLED frozen on the previous
image ‚Äî so they looked identical to "won't boot":

- **`InternalFS.begin()` was never called.** LittleFS then tripped
  `assert(block < block_count)` ‚Üí abort. Fixed with an `ensure_fs()` guard in
  `kv_nrf52.cpp`.
- **nRF52 TWIM I2C has no transaction timeout.** A slave holding SDA low hangs
  `endTransmission` forever. Fixed with `i2c_bus_recover()` plus a `g_oled_ok`
  **headless guard** in `display.cpp`, so a missing/strapped-low OLED doesn't
  wedge boot.

### USB enumerates *before* `setup()` ‚Äî and the banner barely waits

TinyUSB lives in the core, so the port enumerates before the app runs;
enumeration alone never proves `setup()` completed. The USB-CDC boot banner only
waits **500 ms** for a host (`main.cpp`), too tight to attach into after a reset.
To localize a `setup()` hang, temporarily extend the `#ifdef DEVICE_RAK4631`
`!Serial` wait to ~10 s and add breadcrumbs before each init step.

### Reset cause is unreliable on the Adafruit bootloader

The Adafruit nRF52 bootloader **clears `RESETREAS` and wipes GPREGRET2 / `.noinit`
RAM** on the way through, so `platform::reset_reason()` can't natively tell a
watchdog recovery from a power-on. The fix is a **reboot-intent flag in the
flash kv store** (`kv::Store "boot"` key `intent`), written on intentional
`restart()`/`system_off()` and re-armed RUNNING every boot; next boot maps it
onto POWERON / SOFT / WATCHDOG for the bootlog. Register/linker approaches were
tried and **reverted** as too coupled to bootloader internals. Known quirk: the
**RAK spontaneously double-boots on an intentional reboot** (so an operator
`reboot` logs SOFT + a spurious WATCHDOG; a real field wedge is a single clean
WATCHDOG). The Wio does **not** show this quirk. See `watchdog-all-boards` notes.

### BLE notify must be chunked yourself (Bluefruit ‚âÝ NimBLE)

In its default *unbuffered* mode, Bluefruit's `BLEUart::write()` forwards the
whole buffer to one `notify()` of ‚â§ MTU‚àí3 bytes and **silently discards the
rest** while returning success ‚Äî so a long `help`/`bootlog` truncated at ~244 B.
Fix (`ble_provision_nrf52.cpp`): `Bluefruit.configPrphBandwidth(BANDWIDTH_MAX)`
before `begin()`, then **chunk to `getMtu()-3` yourself** with per-chunk
backpressure (retry on `write()==0`) and a single-write `"line\n"` coalesce.
Key difference from the ESP32: NimBLE chunks for you, Bluefruit does not.

### RSSI reads ~41 dB low ‚Äî but RX is not worse

Side-by-side with a Heltec V4 on the same fox, the RAK reads ~41 dB lower RSSI ‚Äî
but the noise floor is ~42 dB lower too, so **SNR is equal** (the RAK was even
~1 dB better, likely its antenna). It's a flat front-end offset (the Heltec's FEM
LNA + RadioLib calibration lift signal *and* noise together), not a sensitivity
deficit. The hunter's RSSI‚Üíloudness/bar mapping is FEM-calibrated, so on the bare
RAK the bar reads cosmetically low; use `rssilog` / SNR to compare. The RAK is
**not** the worse receiver.

### Audio: no onboard sound ‚Äî wire an external amp

The RAK has no speaker hardware. Default sidetone is the same **I2S ‚Üí MAX98357A**
design as the Heltec (`sidetone_nrf52.cpp`, driven via the nRF52840 I2S
registers directly ‚Äî the Adafruit core ships no Arduino I2S wrapper ‚Äî with a
FreeRTOS ping-pong EasyDMA feeder), on the RAK19007 solder headers
(BCLK=17/J11-2, LRCK=31/J11-1, DIN=16/J10-3). Tie the amp's **GAIN ‚Üí VIN**
(floating GAIN ‚Üí intermittent distortion, see `max98357a.md`). I2S master TX is
push-only, so an unwired amp can't hang boot (verified booting Fox with no amp).
A **build-time PWM‚ÜíPAM8403 analog backend** also exists
(`-DSIDETONE_PAM8403 -DPIN_SIDETONE_PWM=16`); the default env stays on the
validated I2S path. Speaker wiring itself is still a hardware TODO.

## Flashing & host tools

```sh
pio run -e rak4631 -t upload --upload-port /dev/cu.usbmodem22301
```

- nrfutil DFU over the **1200 bps touch** auto-reset ‚Äî the RAK firmware
  cooperates, so `pio upload` triggers it without a manual bootloader entry.
- USB IDs: **app VID 0x239A / PID 0x8029**, bootloader PID **0x0029** ("WisCore
  RAK4631 Board"). **Ports renumber** between sessions ‚Äî re-check with
  `scripts/devices.sh --usb` before flashing a named station.
- `devices.sh` shows the nRF52's **64-bit device id** in the CHIP ID column (not
  a MAC); it matches the firmware `show` `chip=` line.
- **nRF52 boards can't be DTR/RTS-reset**, so the HIL test harness only drives
  them as already-running hunters; provision a RAK fox over the **runtime
  console** (`mode`, `keymode`, `reboot` all work live), not by a reset-and-flash.
- If a sibling Wio ever looks "hung at boot," it's usually parked in its UF2
  bootloader, not crashed ‚Äî see `wio-uf2-bootloader-recovery` in the project
  notes.

## Build & variant strategy

- **Vendored variant, stock core.** The board JSON names `variant:
  WisCore_RAK4631_Board`, which the stock Adafruit nRF52 core doesn't ship.
  Rather than pin `platform_packages` to RAKwireless's whole core fork (a much
  bigger, harder-to-pin dependency that replaces the *entire* Arduino core), we
  copy just `variant.h`/`variant.cpp` into `variants/` and point at them with
  `board_build.variants_dir = variants` in `[nrf52_base]`. This keeps the stock
  `framework-arduinoadafruitnrf52` core (no fork-pinning risk) and sidesteps the
  fork's BLE-lockup question ‚Äî revisit `platform_packages` only if that specific
  bug ever shows on hardware. File provenance is in `reference/rak4631/`.
- **The SX1262 is on a dedicated SPI bus, not the Feather header.** The variant's
  default `SPI` (SCK=3 / MOSI=30 / MISO=29) is the Feather/SD bus; our radio
  re-pins the **global `SPI`** to NSS=42 / SCK=43 / MISO=45 / MOSI=44 via
  `SPI.begin(...)` + `new Module(...)` (RadioLib drives CS itself), matching how
  Meshtastic/MeshCore wire the WisCore SX1262.
